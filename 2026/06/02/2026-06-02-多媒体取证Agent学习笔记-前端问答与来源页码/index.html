多媒体取证 Agent 学习笔记：前端问答与来源页码追踪

发表于2026-06-02|更新于2026-06-03|AI
|阅读量:

今天继续推进 Multimedia-Forensics-Agent 项目。

昨天已经在后端跑通了 RAG 基础链路：PDF 上传后可以解析文本、切分 chunks、建立 TF-IDF 检索索引，并通过 /api/papers/ask 返回相关 sources。今天的重点是把这个能力接到 Streamlit 前端，并进一步让 sources 带上页码信息。

今天完成后，项目已经可以在页面上完成：

目前依然没有接入真实大模型，回答部分还是模板式回答，但 RAG 的检索与前端交互链路已经跑通。

1. 今天的目标

今天主要完成：

  • 在 Streamlit 页面接入 /api/papers/ask
  • 使用 st.session_state 保存当前论文的 file_id
  • 理解上传 PDF 用 files=files，提问用 json=payload
  • 增加空问题校验
  • 在前端展示 answer 和 sources
  • 将 VectorStore 的检索结果升级为带 page_number 的 dict
  • 修改上传后的 chunk 构建方式，避免页码丢失

11. 今天踩到的问题

1. 前端状态容易丢

Streamlit 会反复重跑脚本，所以 file_id 必须放到 st.session_state。

否则上传成功后，问答区域可能不知道当前论文是哪一篇。

2. 上传和问答的请求格式不同

问答 JSON 要用：

如果混用，请求就会失败。

3. 先合并全文会丢页码

如果先把所有页面 join 成 full_text，再 split，chunk 就不知道来自哪一页。

所以要逐页 split，并把 page_number 写进每个 chunk item。

当前 answer 还是固定模板，不是大模型生成。

12. 当前阶段总结

今天最大的收获是理解了 RAG 不只是“能搜到文本”，还要能说明“文本来自哪里”。

对于论文研读场景，页码非常重要。因为用户不只是想看模型回答，还需要回到论文原文确认依据。

目前项目已经从：

推进到了：

它已经可以支撑一个基础 Demo。

13. 下一步计划

下一步可以继续做：

  1. 优化 RagEngine 的回答格式，让 answer 根据 sources 自动总结
  2. 接入真实大模型，让 Generate 不再是固定模板
  3. 在前端 sources 中更清晰显示页码和文本
  4. 增加论文结构化分析接口
  5. 开始实现图像指标计算：MSE、PSNR、SSIM

目前最优先的是：让回答从“模板提示”升级成“基于 sources 的简要总结”。
