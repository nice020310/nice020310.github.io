多媒体取证 Agent 学习笔记：Embedding + Chroma 与 LLM 总结

发表于2026-06-06|更新于2026-06-06|AI
|阅读量:

今天继续推进 Multimedia-Forensics-Agent 项目。

前几天已经完成了 PDF 上传、PyMuPDF 文本解析、按页切 chunk、TF-IDF 轻量检索、前端问答和来源页码展示。今天的重点是把项目从 TF-IDF 检索 + 模板回答 升级到更接近真实 RAG 的版本：

这一步很关键，因为项目不再只是根据关键词找片段，也不再只是返回固定模板，而是开始具备“根据论文片段生成回答”的能力。

不过也要诚实说明：目前还没有接入 LlamaIndex Workflow，也还没有接入 OpenAI Agents SDK 工具调用。今天完成的是 Agentic RAG 的基础能力。

1. 今天的目标

今天主要完成了：

  • 理解为什么要从 TF-IDF 升级到 embedding
  • 在 FAISS 和 Chroma 之间选择 Chroma
  • 在 requirements.txt 中加入 chromadb、sentence-transformers 和 openai
  • 改造 app/rag/vector_store.py，使用 embedding + Chroma 做语义检索
  • 改造 app/rag/rag_engine.py，让 RAG 引擎继续复用 vector_store.search()
  • 接入 DeepSeek API，使用 OpenAI-compatible SDK 调用大模型
  • 新增 app/llm/deepseek_client.py，封装 LLM 调用
  • 设计 RAG prompt，要求模型只根据 sources 回答
  • 把 DeepSeekClient 接进 RagEngine，实现 LLM 总结回答
  • 前端继续保持 answer + sources 展示结构不变

2. 为什么要从 TF-IDF 升级到 embedding

之前的 TF-IDF 更像关键词匹配。

它会关注一个词有没有出现、出现多少次。例如用户问：

TF-IDF 不一定能稳定理解 evaluation metrics 和 PSNR / SSIM / BER 之间的关系。

Embedding 更像语义表示。它会把文本变成语义向量，相近含义的句子在向量空间中距离更近。所以 embedding 更适合论文问答场景，尤其是用户问题和论文原文表达不完全一致的时候。

严格来说，TF-IDF 也是向量，但它是基于词频统计的稀疏向量；embedding 是更偏语义的稠密向量。

3. 为什么选择 Chroma 而不是 FAISS

今天比较了 FAISS 和 Chroma。

FAISS 的特点：

  • 本地速度快
  • 更轻量
  • 更偏底层向量索引
  • 需要自己额外管理 metadata

Chroma 的特点：

  • 更像轻量向量数据库
  • 可以同时管理 documents、embeddings、metadatas 和 ids
  • 更适合快速做 RAG Demo
  • 对保存 page_number、text、file_id 更友好

6. 每篇论文使用独立 collection

今天还理解了一个很重要的问题：如果所有论文都用同一个 Chroma collection，就可能出现跨论文污染。

比如用户上传论文 A 后又上传论文 B，如果两篇论文的 chunks 都混在同一个 collection 里，那么针对论文 A 的问题可能检索到论文 B 的内容。

所以更合理的做法是：

每篇论文用自己的 file_id 创建独立 collection。

面试时可以这样解释：

每篇论文的 chunk 和 metadata 应该隔离。用 file_id 创建独立 collection，可以避免用户针对当前论文提问时检索到其他论文内容。

12. 今天最重要的理解

今天最重要的理解有几个：

第一，RAG 不是直接让大模型读整篇论文，而是先检索相关片段，再让大模型基于片段回答。

第二，TF-IDF 和 embedding 都是向量表示，但 TF-IDF 偏关键词统计，embedding 更偏语义理解。

第三，sources 不能只返回 chunk id，而要返回 page_number 和 text。因为 LLM 需要文本作为上下文，用户也需要页码验证来源。

第四，LLM 必须被 prompt 约束。论文问答场景不能编造数据集、指标、实验结果和结论。

14. 面试表达

如果面试官问：

你这个项目今天升级了什么？

可以回答：

我把原来的 TF-IDF 检索层升级成了 embedding + Chroma。原来系统更偏关键词匹配，现在可以用语义向量检索论文 chunk。同时我接入了 DeepSeek LLM，把原来的模板回答升级成基于 sources 的生成式回答。回答 prompt 明确要求只能根据检索片段回答，信息不足时回答不确定，并保留 page_number 做来源追溯。

如果面试官问：

为什么选择 Chroma？

可以回答：

因为当前项目是论文 RAG Demo，需要方便地保存 chunk text、page_number、file_id 等 metadata。Chroma 对文档和 metadata 管理更友好，适合快速做出可展示系统。FAISS 更偏底层和高性能，后续数据量变大时可以考虑。

如果面试官问：

你怎么降低 RAG 幻觉？

可以回答：

我从两层降低幻觉。第一层是检索层，只把 top-k 相关 chunk 作为上下文。第二层是 prompt 约束，要求模型只能根据 sources 回答，如果信息不足必须说不确定，并且不能编造论文中没有出现的数据集、指标、方法和实验结果。同时保留页码，方便用户回到原文验证。

如果面试官问：

为什么前端不用大改？

可以回答：

因为我保持了 /api/papers/ask 的返回结构不变，仍然返回 answer 和 sources。后端内部可以从模板回答升级到 LLM 回答，但 API contract 稳定，所以前端展示逻辑可以复用。
