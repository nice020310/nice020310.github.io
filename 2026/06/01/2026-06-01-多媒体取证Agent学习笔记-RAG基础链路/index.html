多媒体取证 Agent 学习笔记：RAG 基础链路跑通

发表于2026-06-01|更新于2026-06-03|AI
|阅读量:

今天继续推进 Multimedia-Forensics-Agent 项目。

昨天已经跑通了 PDF 上传与 PyMuPDF 文本解析，今天的重点是进入 RAG 的基础链路：把论文文本切成 chunks，用轻量检索方法找到相关片段，并通过 /api/papers/ask 接口完成一次简化版论文问答。

今天还没有接入真实大模型，所以目前不是完整 LLM RAG，而是一个 TF-IDF 检索 + 模板回答 的 MVP。它的价值在于先把 RAG 的数据流跑通。

1. 今天的目标

今天主要完成：

  • 实现 text_splitter.py，把论文长文本切成 chunks
  • 使用 overlap 保留相邻 chunk 的上下文
  • 实现 vector_store.py，用 TF-IDF 做轻量文本检索
  • 实现 rag_engine.py，串起 Retrieve 和简化 Generate
  • 在 paper_api.py 中增加 /api/papers/ask 接口
  • 通过 Swagger 测试 /upload 和 /ask

2. 文本切分：text_splitter.py

RAG 的第一步不是直接上向量数据库，而是先把长文本切成小块。

如果一篇论文很长，系统不能每次都把整篇论文拿去检索或问答。更合理的方式是先切成很多 chunks，然后根据用户问题找到最相关的几个 chunk。

  • overlap：相邻 chunk 之间重叠多少字符
  • 返回值：文本块列表

今天没有直接使用 FAISS 或 Chroma，而是先用 scikit-learn 的 TF-IDF 做轻量检索。

  • 本地容易跑通
  • 适合理解 RAG 的 Retrieve 流程
  • 不会一开始就被复杂向量数据库和 embedding 模型卡住

原因是 TF-IDF 不理解语义，它更像关键词匹配器。它不知道 metrics 和 PSNR / SSIM 是语义相关的概念。

10. 今天踩到的问题

1. TF-IDF 不理解语义

如果问题太抽象，TF-IDF 不一定能返回最符合人类理解的片段。

解决方式是：

  • MVP 阶段尽量用关键词问题测试
  • 后续升级 embedding 检索

2. RagEngine 还不是完整大模型 RAG

3. 内存字典重启会丢失

PAPER_ENGINES 存在内存里，后端重启后会清空。

所以测试时如果重启后端，需要重新上传 PDF。

11. 当前阶段总结

今天最大的收获是理解了 RAG 的基础链路：

RAG 不是一个神秘功能，而是一条数据处理链。

当前版本虽然还没有接大模型，但已经完成了 Retrieve 这一步，并且通过 /ask 接口把问答流程跑通了。

12. 下一步计划

下一步计划把 /ask 接入 Streamlit 前端，让用户不需要 Swagger，也能直接在页面上问论文问题。

同时还要继续增强 sources，让检索结果不仅返回文本，还返回页码。这样后续回答就能追踪到论文具体来源。
