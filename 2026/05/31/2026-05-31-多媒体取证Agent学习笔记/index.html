多媒体取证 Agent 今日学习笔记

发表于2026-05-31|更新于2026-05-31|AI
|阅读量:

今天主要学习和完善的是 Multimedia-Forensics-Agent 这个项目。

这个项目和前几天的私厨 Agent 不太一样。私厨 Agent 更偏生活应用，而这个项目更偏研究辅助：面向数字水印、隐写术、图像取证、多媒体安全论文阅读和实验复现。今天做下来，感觉重点已经从“让模型回答问题”转向“把一个研究型 Agent 项目拆成清晰的工程模块”。

今天完成了什么

今天主要把项目的 MVP 骨架搭起来了。

项目目标是做一个多媒体取证研究助手，后续可以支持：

  • 上传论文 PDF 并提取文本
  • 基于论文内容做问答
  • 分析论文的研究问题、方法、数据集、指标和局限
  • 多篇论文对比
  • 计算图像质量指标，比如 MSE、PSNR、SSIM
  • 生成 Markdown 实验报告

目前已经完成得比较实的是：

  1. FastAPI 后端可以启动。
  2. Streamlit 前端可以连接后端。
  3. 后端拆出了 paper、image、report 三类 API。
  4. PDF 上传接口已经可以保存文件并提取前几页文本。
  5. 用 PyMuPDF 实现了按页读取 PDF 文本。
  6. RAG、图像指标和报告生成模块先留好了接口。

这一步虽然还不是完整 Agent，但项目结构已经比较清楚了。

这个接口做了比较完整的上传流程：

  1. 判断文件名是否存在。
  2. 判断是不是 PDF。
  3. 创建上传目录。
  4. 用 uuid4().hex 生成唯一文件 ID。
  5. 用 Path(file.filename).name 处理文件名，避免路径污染。
  6. 读取上传内容。
  7. 判断文件是否为空。
  8. 保存到本地。
  9. 调用 load_pdf_pages() 提取文本。
  10. 返回 file_id、文件名、页数和前三页文本预览。

3. PDF 解析是 RAG 的前置步骤

今天没有直接上向量库，而是先把 PDF 解析跑通。

这个顺序是对的：先 loader，再 splitter，再 retriever，再 answer。RAG 不是一上来就 FAISS，而是一条数据处理链。

4. 测试脚本也要和代码同步

项目里有几个测试脚本用来验证 PDF 解析和接口逻辑。今天也能看出来，测试脚本如果引用了旧函数名，就会和当前代码不一致。

后面需要把这些临时脚本慢慢整理成正式测试，避免代码能跑、测试却过不了。

和昨天私厨 Agent 的区别

昨天的私厨 Agent 更偏完整应用链路：

  • 用户上传图片
  • OSS 转公网 URL
  • 视觉模型识别
  • 搜索菜谱
  • 流式回答
  • 前端聊天展示

今天这个多媒体取证 Agent 更偏研究工作流：

  • 读论文
  • 提取内容
  • 问答检索
  • 总结分析
  • 对比方法
  • 设计实验
  • 输出报告

这两个项目都叫 Agent，但侧重点不一样。

私厨 Agent 更像产品应用，多媒体取证 Agent 更像科研助手。前者重交互体验，后者重文档理解、分析结构和实验辅助。

今日收获

今天最大的收获是：做 Agent 项目时，领域问题很重要。

如果只是普通聊天，模型本身就能回答很多东西。但一旦进入多媒体取证这种方向，就必须把领域流程拆出来：

  • 论文怎么上传
  • 文本怎么提取
  • 内容怎么切分
  • 检索怎么做
  • 论文结构怎么分析
  • 实验指标怎么计算
  • 最终报告怎么生成

Agent 不只是一个聊天框，而是围绕任务流程组织工具和数据。

今天也更理解了 MVP 的意义：先不急着把所有功能都做完，而是把最小链路跑通。现在已经有了后端、前端、PDF 上传和文本提取，下一步就可以继续往 RAG QA 走。

后续计划

接下来可以按这个顺序继续：

  1. 实现 split_text()，把论文文本切成带 overlap 的 chunks。
  2. 实现一个轻量检索版本，先不用急着接复杂向量库。
  3. 给 /api/papers/ask 加上论文问答接口。
  4. 实现图像质量指标：MSE、PSNR、SSIM。
  5. 把论文分析结果整理成 Markdown 报告。
  6. 最后再考虑 Docker 和更完整的部署。

今天这一步算是把多媒体取证 Agent 的地基打好了。后面要做的，就是让它一点点从“能上传论文”变成“能帮我读论文、比方法、想实验”的研究助手。
