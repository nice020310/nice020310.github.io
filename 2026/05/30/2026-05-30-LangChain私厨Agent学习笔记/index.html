LangChain 私厨 Agent 今日学习笔记

发表于2026-05-30|更新于2026-05-30|AI
|阅读量:

今天把昨天的 LangChain notebook 实战继续往前推了一步。

昨天主要还是在 notebook 里面跑通模型、工具、记忆和多模态输入。今天开始把它拆成一个真正的小项目：后端用 FastAPI，Agent 用 LangGraph，前端用 Next.js，再加上 OSS 图片上传，最后形成一个可以聊天、上传食材图片、流式返回菜谱建议的 AI 私人厨师应用。

今天完成了什么

今天的目标不是单纯让模型回答问题，而是把完整链路跑起来：

  1. 前端可以输入文字，也可以上传食材图片。
  2. 图片先上传到 OSS，得到一个模型可以访问的图片 URL。
  3. 后端接收文字、图片 URL 和会话 ID。
  4. Agent 使用视觉模型识别图片，再调用搜索工具查菜谱。
  5. 回答通过流式接口一点点返回到前端。
  6. 同一个 thread_id 可以保留历史对话。
  7. 用户可以新建会话，也可以清空历史记录。

到这里，项目就不再只是一个 notebook demo，而是有了应用的形状。

2. 旧的 thread_id 会保留错误历史

之前同一个 thread_id 里存过错误格式的图片消息，后面即使代码改对了，Agent 仍然会把旧消息带上，继续报错。

解决方式有两个：

  • 换一个新的 thread_id
  • 或者清空这个会话的历史记录

这也让我更直观地理解了 checkpointer：它是真的在保存状态，不是摆设。

今日收获

今天最大的收获是：Agent 应用不是只写 prompt。

一个真正能用的小应用，需要把很多环节串起来：

  • 模型选择
  • 多模态消息格式
  • 工具调用
  • 会话记忆
  • 流式输出
  • 图片上传
  • 前后端接口
  • 静态资源部署
  • 错误处理和日志

之前在 notebook 里看到的是“模型能回答”，今天更像是在做“模型怎么接进产品”。

LangChain 和 LangGraph 最有意思的地方也在这里：它们不是单独替我完成应用，而是把模型、工具、状态和调用流程组织起来。真正写项目的时候，还是要考虑接口、数据结构、前端体验和部署方式。

今天这个 AI 私人厨师项目已经有了雏形，后面可以继续完善：

  • 让推荐结果更结构化
  • 给菜谱卡片增加图片和评分
  • 优化历史消息里多模态内容的展示
  • 增加错误提示和加载状态
  • 部署后端，让前端不只跑在本地

总之，今天算是从“会调用模型”往“会做 AI 应用”迈了一步。
